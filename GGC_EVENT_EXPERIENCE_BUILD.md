# Groomed Gent Co. Event Experience — Codex Build Sequence

## Mission

Build a permanent, production-ready event-display experience for Groomed Gent Co. It will run full-screen on laptops, tablets, and phones at farmers markets, pop-ups, expos, barbershop events, and future retail displays. The experience must look like masculine modern luxury while making the real QR code effortless to scan from several feet away.

The primary logo is already available at the project root as `logo.png`.

## Operating instructions for Codex

Codex may execute every prompt in this document in order without waiting for confirmation between prompts. Treat each numbered prompt as a separate, small implementation task.

For every prompt:

1. Inspect the current repository and preserve all existing valid work.
2. Implement only the requested scope; avoid unrelated refactors.
3. Run the smallest meaningful validation for that step.
4. Fix failures caused by the step before continuing.
5. Create or update `BUILD_STATUS.md` immediately after the prompt. Record the prompt number and title, status, files changed, validation run and result, decisions or assumptions, and any remaining follow-up.
6. Continue to the next prompt only after the current step is complete and minimally validated. If genuinely blocked, record the exact blocker in `BUILD_STATUS.md` and stop rather than inventing credentials, prices, product facts, or external configuration.

Use the project's existing package manager, framework conventions, linting, formatting, and component patterns when present. Never overwrite valid configuration merely to match a preferred template. Keep dependencies lean. Do not commit secrets. Do not commit or push unless explicitly asked.

## Fixed product and brand requirements

- Brand: **Groomed Gent Co.**
- Experience name: **Groomed Gent Co. Event Experience**
- CTA: **SCAN TO SHOP**
- Brand statement: **Where Grooming Meets Health. Where Mindset Builds Legacy.**
- Offer: **15% off your next online order**
- Coupon: **LEGACY15**
- Store: `https://groomedgentco.com`
- Shopify discount destination: `https://groomedgentco.com/discount/LEGACY15?redirect=%2F`
- Legacy Gold: `#C4912F`
- Palette: obsidian black, graphite, deep imperial purple, warm ivory, dimensional restrained gold
- Avoid bright yellow gold, cartoon styling, cheap gradients, excessive glow, clutter, tiny text, fake 3D, and distracting motion.

The Shopify discount must also be configured and active in Shopify Admin; the app can generate the correct discount URL but cannot create the store-side discount rule without authorized store access. Never claim the offer works until the destination is tested against the configured Shopify discount.

## Target architecture and content model

Prefer a single responsive App Router page composed from focused components and typed data:

- `EventExperience`: page composition and mode state
- `AmbientBackground`: decorative, non-interactive visual environment
- `BrandHeader`: `logo.png`, brand name, and statement
- `ScanPanel`: CTA, real QR, offer, coupon, and fallback URL
- `ProductCatalog`: data-driven product/pricing cards
- `DisplayControls`: display/catalog switch, full-screen, and optional wake-lock control
- `products` data module: the single easy-to-edit source for names, descriptions, availability, and prices
- small hooks/utilities only where browser APIs or reusable behavior require them

Names and file locations may follow a valid existing project convention. Keep the QR value, store URL, coupon, offer copy, and product data centralized rather than duplicated across components. Default to **Display mode**: the primary CTA and QR should fit without scrolling at common laptop, tablet, and phone sizes. Catalog content can appear below the fold and/or through a clean mode switch.

## Definition of done

- The QR is generated by a real QR library as SVG or canvas, never drawn or generated by AI.
- The QR points exactly to the Shopify discount URL and remains stationary, high contrast, and unobstructed inside a generous light quiet zone.
- `logo.png`, CTA, offer, coupon, fallback URL, and brand statement are clearly visible.
- Current pricing is rendered from one typed, easy-to-update data source. Unknown prices remain explicit placeholders or are omitted; they are never invented.
- Display and catalog behavior is clean across phone, tablet, and laptop viewports.
- Full-screen control works where supported; screen wake lock is progressive enhancement with clear unsupported/error states.
- Reduced-motion, keyboard use, focus visibility, semantic structure, contrast, and screen-reader needs are handled.
- Decorative motion never moves, distorts, flashes, overlays, or reduces the scanability of the QR.
- Production build, lint/type checks, route checks, and QR destination verification pass.

## Prompt 02 — Establish tokens, architecture, and centralized content

Create the focused page/component structure described above, adapted to existing conventions. Add typed centralized configuration for the store URL, Shopify discount URL, `LEGACY15`, offer copy, CTA, brand statement, and presentation-safe labels. Establish CSS design tokens for obsidian, graphite, imperial purple, warm ivory, `#C4912F`, restrained dimensional gold, spacing, radii, shadows, and typography. Render a semantic structural skeleton with no elaborate motion yet. Validate type checking and linting.

## Prompt 03 — Implement and verify the real discount QR

Add a small, reputable QR library compatible with the existing stack and render a real code-generated QR for exactly `https://groomedgentco.com/discount/LEGACY15?redirect=%2F`. Prefer SVG for crisp resizing. Use very dark modules on a white or warm-ivory square, a generous standards-safe quiet zone, high error correction where supported, and no embedded logo or decoration. Keep the QR stationary. Add the visible fallback `groomedgentco.com` and coupon `LEGACY15`. Add a focused test or deterministic assertion proving the encoded value comes from centralized configuration. If network access is available, verify the URL resolves without embedding secrets; otherwise record manual destination testing as pending.

## Prompt 04 — Build the distance-readable display layout

Implement the primary display viewport. Use `logo.png` prominently with correct intrinsic sizing and accessible alt text. Make **SCAN TO SHOP**, the QR, `LEGACY15`, “15% off your next online order,” the fallback URL, and the brand statement legible and visually ordered. Optimize the default view for several-feet-away viewing, strong contrast, large type, generous whitespace, and minimal clutter. Ensure the QR is not pushed below the fold at representative viewport sizes. Validate at phone, tablet, and laptop dimensions.

## Prompt 05 — Add easy-to-update product and pricing data

Create a typed product data module as the single source for product name, short description, price label or price in cents, availability, optional badge, and display order. Inspect the repository for authoritative product/pricing information before populating it. Do not invent missing prices or claims: use clearly marked placeholders in code and a restrained “Ask us for today’s pricing” presentation where necessary. Build accessible product cards/catalog content from this data and add a short maintainer comment or README note explaining exactly where prices are updated. Validate rendering for zero, one, and several products.

## Prompt 06 — Apply the premium visual system

Refine the page into an obsidian/graphite environment with deep imperial-purple depth, warm-ivory typography, Legacy Gold accents, thin architectural lines, and restrained dimensional details. Make the logo feel integrated and premium without altering the source asset. Avoid bright yellow, heavy bloom, fake metallic text effects, excessive gradients, visual noise, and low-contrast copy. Keep all decoration outside the QR quiet zone. Validate contrast and inspect the complete visual hierarchy at common viewports.

## Prompt 07 — Add subtle scan-safe motion

Add lightweight ambient motion around—not on—the QR: for example slow atmospheric drift, restrained edge-line movement, or a subtle logo breathing treatment. Use CSS animations where practical and avoid a heavy animation dependency unless the existing stack already uses one. The QR, quiet zone, CTA, coupon, and fallback URL must never translate, rotate, scale, flicker, blur, pulse, or receive an animated overlay. Pause or minimize ambient work when the page is hidden. Validate smoothness and confirm scan-critical elements remain pixel-stable.

## Prompt 08 — Perfect responsive behavior

Tune layout and type with fluid sizing and intentional breakpoints for small phones, large phones, tablets in both orientations, and laptop displays. Account for safe-area insets and modern dynamic viewport units. Prevent clipping, horizontal overflow, unreadably small offer text, and an undersized QR. In Display mode, prioritize logo, CTA, QR, coupon, offer, and fallback URL above optional content. Validate at least 375×667, 390×844, 768×1024, 1024×768, and 1440×900.

## Prompt 09 — Add Display and Catalog modes

Implement a clean, keyboard-accessible mode switch with **Display** as the default and **Catalog** as the secondary view. Display mode should be calm, distance-readable, and QR-first. Catalog mode should show product/pricing content without losing an obvious route back to the QR. Keep state local unless an existing routing pattern makes query parameters useful; do not add unnecessary navigation. Ensure the control has correct semantics, visible focus, selected state, and touch-friendly targets. Validate switching with pointer and keyboard.

## Prompt 10 — Add full-screen and wake-lock controls

Add a compact full-screen control using the Fullscreen API with feature detection, accurate enter/exit state, and graceful error handling. Add an optional “Keep screen awake” control using the Screen Wake Lock API where supported; request it only after a user gesture, release it cleanly, and attempt reacquisition after visibility changes only when the user’s preference remains enabled. Present clear supported, active, inactive, and error/permission states without blocking the experience. Hide or disable unsupported controls appropriately. Validate controls with mocked APIs or focused tests plus a manual-test note.

## Prompt 11 — Accessibility and reduced motion

Audit semantic landmarks, heading order, descriptive labels, image alt text, keyboard navigation, touch target sizes, focus indicators, contrast, and status announcements for browser controls. Honor `prefers-reduced-motion: reduce` by eliminating nonessential animation rather than merely shortening it. Mark ambient visuals as decorative and prevent them from intercepting input or reaching assistive technology. Ensure zoom and text resizing do not break the page. Run the available accessibility checks and record results.

## Prompt 12 — Performance and resilience pass

Optimize the local logo through the framework image pipeline where appropriate, avoid layout shift, keep client components narrowly scoped, and remove unnecessary dependencies or effects. Ensure the core logo, offer, fallback URL, and code-generated QR render reliably without waiting on third-party scripts or remote media. Handle JavaScript/browser-API failures gracefully. Check bundle/build output and fix material warnings. Confirm the experience remains useful if motion and optional browser APIs are unavailable.

## Prompt 13 — Metadata and installable display polish

Add accurate page title, description, theme color, viewport behavior, icons only when valid local assets exist, and suitable social metadata without inventing an image URL. If appropriate for the existing stack, add a minimal web app manifest with standalone display intent, brand colors, and correct names. Do not fabricate icon files by renaming incompatible assets. Add only standards-based PWA-style polish that works on Vercel without introducing a fragile service worker. Validate manifest/metadata output.

## Prompt 14 — Automated QR and UI safeguards

Add lean automated coverage using the project’s existing test tools, or the smallest justified test setup if none exists. Cover centralized discount URL and coupon values, visible CTA/offer/fallback copy, product-data rendering, reduced-motion behavior where testable, and browser-control feature detection. Add an automated QR safeguard that reads or decodes the rendered QR when practical; otherwise assert the QR component receives the exact destination and record a required physical scan test. Do not add a large end-to-end stack solely for trivial assertions.

Perform a final complete-system test at the specified phone, tablet, and laptop sizes. Verify no overlap, clipping, overflow, layout shift, or scan obstruction in either mode. On at least two physical phones with different camera/OS combinations, scan the QR from the laptop/tablet display at multiple practical distances and confirm it opens the exact Groomed Gent Co. Shopify discount destination. Record all results and remaining external actions in `BUILD_STATUS.md`.

## Final handoff

After Prompt 16, make `BUILD_STATUS.md` a concise source of truth containing:

- overall build status;
- completed prompts and validations;
- exact production commands run;
- authoritative product/pricing data still needed, if any;
